# Bilingua 2a — Static UI & Admin → English-primary (design spec)

**Дата:** 2026-06-22
**Направление:** 2a (из 3: Ремедиация ✅ → **Билингва 2a (UI/админка)** → Билингва 2b (контент) → Конструктор)
**Статус:** на ревью

## Контекст

`d:\retro` — Django 5.2 (`distributor/`) + Vite/Tailwind мульти-вертикальный сайт. Сейчас:
- `LANGUAGE_CODE = "uk"`, `MODELTRANSLATION_DEFAULT_LANGUAGE = "uk"`, fallback `("uk","en")`, языки `uk, en`.
- Исходные строки в коде — **украинские**; английский лежит переводом в `locale/en/LC_MESSAGES/django.po`
  (354 msgid, ~11 без перевода). Локали `uk` нет.
- `modeltranslation` подключён только к `SiteSettings` (24 поля-лейбла, включая `brand_name`).
- Метки админки/Jazzmin — украинские (часть через gettext, Jazzmin — хардкод).
- Переключатель языка EN/UK в навбаре уже есть (`set_language`).

Это направление 2 разбито пользователем на **2a (статический UI + админка)** — данный спек — и **2b (контент)** —
отдельный цикл. 2a сознательно не трогает перевод контентных моделей и их данные.

## Цель

Сделать **английский основным языком** сайта и админки, с украинским как переключаемым вариантом,
через **классический gettext-подход (Path A): исходные строки в коде — английские**, а украинский —
в `locale/uk`. «Нет украинского значения → показываем английское» обеспечивается fallback'ом.

## Не-цели (вне 2a)

- Перевод контентных моделей (Product/Category/Brand/News/Promo/Vacancy/Banner) и миграция их данных → **2b**.
- Любые изменения визуального конструктора → направление 3.

## Подход (выбран): Path A — английский source

Следствие: при английском source **английскому каталог перевода не нужен** (gettext возвращает msgid).
Нужен только `locale/uk`. Текущий `en.po` (укр.msgid → англ.msgstr) — **источник данных** для генерации
`uk.po` инверсией, после чего `locale/en` удаляется.

## Дизайн

### 1. Поток переводов
1. Переписать все строки-источники в коде на **английский**: `_()`, `{% trans %}`, `{% blocktrans %}`,
   `|default:_("...")`, `verbose_name`/`verbose_name_plural`/`help_text`, метки `TextChoices`,
   лейблы/плейсхолдеры/ошибки форм, сообщения валидаторов, строки Jazzmin (обернуть в `gettext_lazy`).
2. Сгенерировать `locale/uk/LC_MESSAGES/django.po` (`makemessages -l uk`).
3. Заполнить `uk.po` украинскими переводами. Источник: **инверсия текущего `en.po`** —
   для нового англ. msgid украинский перевод = старый украинский source (старый msgid).
   Для ~11 строк без английского: автор английского source вручную, украинский = старый source.
4. Удалить `locale/en` (английскому каталог не нужен).
5. `compilemessages` (уже в старте dev-контейнера) должен проходить.

### 2. Настройки (`config/settings/base.py`)
- `LANGUAGE_CODE = "en"`.
- `LANGUAGES = [("en", "English"), ("uk", "Українська")]` (английский первым).
- `MODELTRANSLATION_DEFAULT_LANGUAGE = "en"`.
- `MODELTRANSLATION_LANGUAGES = ("en", "uk")`.
- `MODELTRANSLATION_FALLBACK_LANGUAGES = ("en", "uk")` — пустое значение на активном языке
  подхватывает другой, чтобы ни EN, ни UK не показывали пусто.

### 3. SiteSettings (входит в 2a, т.к. её поля — UI-лейблы)
- Дефолты переводимых полей в коде: украинский → **английский** (`default="Каталог"` → `default="Catalog"` и т.д.).
- **Исключение «название сайта»:** `brand_name` убирается из `apps/core/translation.py` → становится
  **одним полем** (без `_en/_uk`). Схемная миграция удаляет `brand_name_en/brand_name_uk`,
  сохраняя `brand_name` (текущее значение). Остальные поля остаются двуязычными.
- **Data-миграция singleton** (после флипа default на `en`): для каждого оставшегося переводимого поля
  сохранить текущее украинское значение в `*_uk`; засеять `*_en` английским значением
  (английский дефолт поля); привести базовое поле к `*_en` (modeltranslation держит base == default-язык).
  Так EN-основной не пустой, UK сохранён.

### 4. Объём правок по слоям
- **Модели** (`apps/*/models.py`): verbose_name(_plural), help_text, метки TextChoices.
- **Админка** (`apps/*/admin.py`): заголовки/описания филдсетов, `short_description`, `PRESET_LABELS`,
  кастомные лейблы (`core/admin.py` и др.).
- **Шаблоны** (`templates/**`): все `{% trans %}`/`{% blocktrans %}` и украинские `|default:_("...")`.
- **Формы** (`contacts/forms.py`, `orders/forms.py`): labels, placeholders, error messages.
- **Валидаторы** (`apps/core/validators.py`): сообщения `validate_contactable`, `validate_image_size`.
- **Jazzmin** (`config/settings/base.py`): `site_title`, `site_header`, `site_brand`, `welcome_sign`,
  `copyright` → `gettext_lazy` с английским source.

### 5. Переключатель языка
- Существующий переключатель EN/UK сохраняется; порядок кнопок — EN первой (соответствует `LANGUAGES`).

## Тестирование
- Активный язык `en` → ключевые строки рендерятся по-английски (вьюха/шаблон отдают «Catalog», не «Каталог»).
- Активный язык `uk` → строки рендерятся по-украински (из `uk.po`).
- SiteSettings после миграций: `*_en` заполнены; `brand_name` — одиночное поле (нет `_en/_uk`);
  fallback: пустой `_uk` → показывает `_en`.
- `compilemessages` без ошибок; `uk.po` без фаззи/пустых на критичных строках.
- Полный сьют (`manage.py test`) и `manage.py check` зелёные.
- Тесты в Docker: `docker compose -f docker-compose.dev.yml exec -T web python manage.py <cmd> --settings=config.settings.development`.

## Миграции
- `core`: схемная (удаление `brand_name_en/_uk`) + data-миграция (засев `*_en`, сохранение `*_uk`).
- Прочие приложения: изменения строк (verbose_name/help_text) генерируют косметические миграции —
  включить их, чтобы `makemigrations --check` был чист.

## Критерии приёмки
- Дефолтный посетитель и админ видят **английский**; переключение на украинский работает.
- Нет регрессий: полный сьют и `check` зелёные; `makemigrations --check` без хвостов.
- `brand_name` — одиночное; остальные SiteSettings-лейблы двуязычны с английским по умолчанию.
- В коде исходные строки английские; украинский — в `locale/uk`; `locale/en` удалён.

## Риски
- `{% blocktrans %}` с переменными и `%(...)s`-интерполяции: аккуратно переносить плейсхолдеры
  (несовпадение ломает перевод/рендер).
- Большой механический дифф: бить на изолированные задачи по приложениям/слоям, каждая с тестом
  и прогоном `compilemessages`/сьюта.
- Инверсия `en.po`: сверять, что каждый новый англ. msgid получил украинский перевод; фаззи-записи вычистить.
- modeltranslation base/default-консистентность при флипе: data-миграция должна выставлять base = `*_en`.
