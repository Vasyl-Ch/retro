# Bilingua 2b — Bilingual content (modeltranslation) — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make catalog/content/vacancies content fields bilingual (`_en`/`_uk` via django-modeltranslation), migrate existing Ukrainian content into the `_uk` variant, give editors tabbed EN/UK admin, and sanitize rich-text in every language.

**Architecture:** Register `TranslationOptions` per app → modeltranslation auto-adds `*_en`/`*_uk` columns (schema migration). A per-app data migration copies the existing base value into `*_uk` via raw SQL (leaving `*_en` empty; fallback `en→uk` keeps the site populated). Admins switch to `TabbedTranslationAdmin` + `Translation*Inline`; rich-text `save()`-sanitization is extended to all language variants.

**Tech Stack:** Django 5.2, django-modeltranslation (`TabbedTranslationAdmin`, `TranslationStackedInline`, `TranslationTabularInline`), django-summernote, bleach.

## Global Constraints

- **Settings already (from 2a):** `LANGUAGE_CODE="en"`, `MODELTRANSLATION_DEFAULT_LANGUAGE="en"`, `MODELTRANSLATION_LANGUAGES=("en","uk")`, `MODELTRANSLATION_FALLBACK_LANGUAGES=("en","uk")`. Do not change these.
- **Translatable fields (exact):** catalog `Brand(name,description)`, `Category(name)`, `Product(name,description,location)`, `VehicleSpec(color)`, `ProductSpec(label,value)`; content `Banner(title,subtitle,button_text)`, `News(title,preview,content)`, `Promo(title,description)`; vacancies `Vacancy(title,city,short_tagline,description,requirements,conditions)`, `VacancyImage(caption)`.
- **Not translated:** `slug` (single, URL-stable), `article`, choice fields, `button_url`, `vin`, prices, images; user data/snapshots (`Order`, `OrderItem.product_name`, `ContactRequest`).
- **Existing content → `_uk`; `_en` left empty.** Do NOT run `manage.py update_translation_fields` (it would copy base→`_en`). Use the custom raw-SQL data migration in Task 2.
- **No new dependencies** (modeltranslation/summernote/bleach already installed).
- **Tests in Docker:** `docker compose -f docker-compose.dev.yml exec -T web python manage.py <cmd> --settings=config.settings.development` (run from `d:\retro`; app labels, no `distributor.` prefix). Dev stack already running — do not rebuild.
- **Rich-text test gotcha:** `SummernoteTextField` masks disallowed tags on DB READ, so assert on the **in-memory** instance (no `refresh_from_db()`) — see the Bilingua 2a sanitization tests for the pattern.
- Branch: `bilingua-2b` (already checked out).

**Table names (for raw SQL):** `catalog_brand`, `catalog_category`, `catalog_product`, `catalog_vehiclespec`, `catalog_productspec`, `content_banner`, `content_news`, `content_promo`, `vacancies_vacancy`, `vacancies_vacancyimage`.
```python
"""Shared raw-SQL backfill: copy each base column into its `_uk` variant.

Raw SQL avoids modeltranslation/SummernoteTextField field machinery in
historical migration models and copies exact stored bytes."""


def backfill(connection, table_fields):
    with connection.cursor() as cursor:
        for table, fields in table_fields:
            for field in fields:
                cursor.execute(
                    f"UPDATE {table} SET {field}_uk = {field} WHERE {field}_uk IS NULL"
                )
```

```python
def sanitize_instance_html(instance, *field_names, languages=("en", "uk")):
    """Sanitize rich-text fields on `instance`, including modeltranslation
    language variants (<field>_<lang>) when those attributes exist."""
    for base in field_names:
        for name in (base, *(f"{base}_{lang}" for lang in languages)):
            if hasattr(instance, name):
                value = getattr(instance, name)
                if value:
                    setattr(instance, name, sanitize_html(value))
```
